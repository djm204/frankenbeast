# @franken/web Ramp-Up

**Status**: **INTEGRATED (Frontend)** — This is the primary human-facing dashboard for the Frankenbeast framework. It connects to the `@franken/orchestrator` through same-origin HTTP/WebSocket paths, usually via the Vite dev proxy in local development.

## Module Overview

A React-based single-page application (SPA) that provides a control plane for operators. It lets users chat with the agent, launch and monitor tracked Beast agents, manage local network services, inspect analytics/cost/safety telemetry, and configure dashboard-facing provider, security, and skill controls.

## Current Functionality

- **Overview dashboard**: Skill catalog, provider, and security panels backed by dashboard snapshot APIs.
- **Live Chat**: Real-time interaction via the orchestrator `chat-server` WebSocket and chat session REST APIs.
- **Tracked Agents**: Launch, inspect, edit, and control long-running Beast agents and linked runs.
- **Network Controls**: View network mode/secure backend, start/stop/restart services, edit assignments, and inspect service logs.
- **Analytics**: Browse observer/governor/security/cost telemetry, session filters, summaries, and event details.

## Integration Details

- **Backend**: Communicates with the `@franken/orchestrator` HTTP server, defaulting to `http://127.0.0.1:3737` during local Vite proxy workflows.
- **Authentication**: Uses the server-side `FRANKENBEAST_BEAST_OPERATOR_TOKEN` generated by `frankenbeast init`. Put it in the repo root `.env` for local development so both the orchestrator and Vite proxy can read it. Do **not** use `VITE_BEAST_OPERATOR_TOKEN` or any `VITE_*` variable for the long-lived operator token because Vite exposes `VITE_*` values to browser code.
- **Protocol**: Browser code calls same-origin `/v1/*` and `/api/*` paths. Vite proxies those paths to the backend and injects the operator token server-side only; chat uses the proxied WebSocket endpoint.

## Dashboard Entry Points (src/)

- `app.tsx`: Resolves the same-origin base URL and mounts the dashboard shell.
- `components/chat-shell.tsx`: Main dashboard shell, route tabs, shared clients, chat session wiring, and page composition.
- `hooks/use-chat-session.ts`: Chat WebSocket/session state, reconnect behavior, transcript updates, and chat error banners.
- `pages/dashboard-page.tsx`: Overview dashboard for skills, provider, and security panels.
- `pages/beasts-page.tsx`: Tracked-agent launch/detail/control surface, backed by `components/beasts/*` wizard, list, action, and detail components.
- `pages/network-page.tsx`: Network service controls, config editor, status grid, and logs panels.
- `pages/analytics-page.tsx`: Analytics summaries, filters, event list, and event detail dialog.
- `components/providers/provider-panel.tsx`, `components/security/security-panel.tsx`, and `components/skills/skill-catalog-browser.tsx`: Current provider/security/skills panels surfaced on the Overview route.

## Typed API Clients (src/lib/)

- `lib/api.ts`: Chat REST client, session listing, transcript fetches, and typed chat/session payloads.
- `lib/beast-api.ts`: Beast run and tracked-agent clients, catalog types, lifecycle actions, and log streams.
- `lib/network-api.ts`: Network status, service control, config, and log APIs.
- `lib/analytics-api.ts`: Analytics summary, event, session option, and detail APIs.
- `lib/dashboard-api.ts`: Overview dashboard snapshot and mutation APIs for skills, security, and providers.
- `lib/resolve-base-url.ts`: Browser-safe same-origin base URL resolution.

## Local Development Workflow

From the repo root, start the orchestrator chat server for the project you want the dashboard to inspect:

```bash
npm --workspace @franken/orchestrator run chat-server -- --base-dir /path/to/your-project
```

Then start the dashboard with the chat-server proxy:

```bash
npm --workspace @franken/web run dev:chat
```

Open the Vite URL, usually `http://127.0.0.1:5173/`. The `dev:chat`, `dev`, and `preview` workflows keep browser requests same-origin and proxy `/api` and `/v1` to the backend.

For a non-default backend port, leave `VITE_API_URL` unset and set the proxy target instead:

```bash
VITE_API_PROXY_TARGET=http://127.0.0.1:4242 npm --workspace @franken/web run dev
```

Set `VITE_BEAST_API_PROXY_TARGET` only when Beast controls under `/v1/beasts/*` must talk to a different local Beast daemon than the chat/API backend.

## Build & Test

```bash
npm --workspace @franken/web run dev        # Start Vite dev server
npm --workspace @franken/web run dev:chat   # Start Vite with chat-server proxy defaults
npm --workspace @franken/web run build      # Production build (tsc + vite build)
npm --workspace @franken/web run typecheck  # TypeScript check for the web package
npm --workspace @franken/web test           # Vitest tests
```
