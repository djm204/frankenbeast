# Work Command Design

> Date: 2026-03-10
> Status: Approved
> Scope: `frankenbeast work`, issue fallback, plan frontmatter standardization, bounded plan review/improvement loops, isolated worktree execution, and plan authoring support

## Problem

Frankenbeast has useful pieces of a work pipeline, but they are split across separate commands and inconsistent artifacts:

- `interview`, `plan`, `run`, and `issues` expose phases rather than a single end-to-end workflow
- implementation plans are not discoverable by a deterministic artifact contract
- chunk files and parent plan files are both Markdown, but the CLI has no canonical way to distinguish them
- plan review is not yet strong enough to answer "is this runnable?" with principal-level scrutiny
- issue-driven work is a separate path instead of a fallback source under one unified workflow
- current execution isolation is branch-oriented, but `work` needs explicit worktree-based isolation so `main` stays usable
- people cannot easily author or retrofit their own plans for `work` to discover

The missing product is a first-class command that can find work, create missing plans, improve and vet them repeatedly, then execute safely.

## Goal

Add a first-class `frankenbeast work` lifecycle that:

1. Discovers local implementation plans via frontmatter
2. Falls back to GitHub issues when no runnable plan exists
3. Verifies issue freshness against the current codebase before acting
4. Creates or repairs plans as needed
5. Runs bounded `improve -> vet -> improve` loops with structured scoring
6. Executes only accepted or `good enough` work
7. Runs all execution in isolated git worktrees on feature branches
8. Teaches users how to author plans and retrofit existing Markdown with a supported CLI command

## Non-Goals

- replacing `interview`, `plan`, or `run` as standalone lower-level commands
- making every Markdown file in the repo runnable work
- inventing a new module outside `franken-orchestrator` for this workflow
- auto-resolving ambiguous chunk lineage when metadata is missing
- infinite self-improvement loops

## User Workflows

### Default work flow

```bash
frankenbeast work
```

Behavior:

1. Discover local implementation plans
2. Rank them by priority, status, and freshness
3. If none are runnable, fall back to open issues
4. Create or improve a plan
5. Vet and score it
6. Execute in an isolated worktree on a feature branch

### Issue-forced work flow

```bash
frankenbeast work issues
frankenbeast work issues --label bug,security
```

Behavior:

- `frankenbeast work issues` forces issue-sourced work
- `--label` remains optional
- if `--label` is omitted, the workflow scans open issues in `critical -> high -> medium -> low` order

### Compatibility alias

```bash
frankenbeast issues
```

This remains as a compatibility alias to `frankenbeast work issues`.

### Plan authoring / retrofitting

```bash
frankenbeast plan prepare ./docs/plans/my-plan.md
frankenbeast plan prepare ./docs/plans ./notes/roadmap.md
```

Behavior:

- accepts one or more file or directory paths
- walks directories for Markdown files
- adds required frontmatter when missing
- merges required keys into existing frontmatter without deleting unrelated keys
- skips ambiguous chunk files rather than guessing unsafe lineage

## Canonical Artifact Contract

Any Markdown file is a discoverable implementation plan only if its frontmatter contains `isImplementationPlan: true`.

### Parent implementation plan frontmatter

```yaml
---
isImplementationPlan: true
planId: "2026-03-10-work-command"
status: "draft"
source: "local"
priority: "high"
score: 0
maxScore: 100
---
```

Required semantics:

- `isImplementationPlan: true` marks a top-level runnable plan artifact
- `planId` is the stable identity used by state, scoring, and resume
- `status` supports at least `draft | vetted | runnable | running | completed | superseded`
- `source` supports at least `local | generated-from-issue`
- `priority` supports `critical | high | medium | low`
- `score` and `maxScore` capture the most recent review outcome

### Chunk frontmatter

```yaml
---
isChunk: true
chunkId: "wire-plan-discovery"
implementationPlan: "docs/plans/2026-03-10-work-command-implementation-plan.md"
status: "draft"
score: 0
maxScore: 100
---
```

Required semantics:

- `isChunk: true` excludes the file from top-level runnable plan discovery
- `chunkId` is the stable chunk identity
- `implementationPlan` is the canonical back-reference to the parent plan file
- chunk status and scoring are tracked separately from the parent plan

## Planner Output Rules

The planner must treat frontmatter emission as a hard requirement:

- any implementation plan generated by Frankenbeast must include `isImplementationPlan: true`
- any chunk emitted from a parent plan must include `isChunk: true`
- every chunk must include `implementationPlan` pointing at its parent implementation plan path

This should be enforced by output writers, not by prompt hope alone. Prompt instructions should still explicitly require the frontmatter, but the writer remains the final source of truth.

## CLI Surface

### New top-level command

```bash
frankenbeast work
frankenbeast work issues
```

### New nested planning utility

```bash
frankenbeast plan prepare <paths...>
```

### Compatibility behavior

- `frankenbeast issues` aliases to `frankenbeast work issues`
- existing `interview`, `plan`, and `run` remain available as lower-level commands

## Pipeline

Recommended owner: `WorkCoordinator` in `packages/franken-orchestrator/src/work/`.

### Stage 1: discover

`PlanDiscoveryService` scans repo Markdown, parses frontmatter, excludes ignored directories, and returns candidate parent plans and chunks.

Ignored directories should include at minimum:

- `.git`
- `node_modules`
- `.worktrees`
- `.frankenbeast/.build`

Selection rules:

- include only files with `isImplementationPlan: true`
- exclude files with `isChunk: true`
- rank by `priority`, `status`, and recency

### Stage 2: source

If discovery returns no runnable local plan, or the user explicitly requested issues, `IssueWorkSource` becomes the source of work.

If `--label` is omitted:

1. fetch `critical`
2. if none remain actionable, fetch `high`
3. then `medium`
4. then `low`

Before accepting an issue, `IssueFreshnessVerifier` must check whether the issue is already fixed, stale, duplicated by a local plan, or otherwise no longer actionable against the current codebase.

### Stage 3: create

If an actionable issue has no suitable implementation plan, `PlanSynthesisService` creates a new implementation plan Markdown file with required frontmatter and issue lineage metadata.

Issue-derived plans should record:

- `source: generated-from-issue`
- issue number / URL in frontmatter or a dedicated metadata block

### Stage 4: chunk

`ChunkingService` determines whether the plan is mature enough to run as a whole or should be decomposed into chunks first. If chunked:

- chunk files receive required chunk frontmatter
- the parent implementation plan remains the canonical top-level artifact
- chunk lineage always points back via `implementationPlan`

### Stage 5: vet

`PlanReviewOrchestrator` runs structured expert review and produces:

- per-reviewer findings
- per-category scores
- weighted composite score
- verdict: `accept | improve | good_enough | reject`

### Stage 6: improve

`PlanImprover` revises the plan or chunk set from structured findings rather than from a vague freeform retry prompt.

### Stage 7: bounded loop

The workflow repeats `vet -> improve` up to a hard maximum of 5 iterations.

Decision rules:

- if `score >= acceptanceScore`, mark runnable and proceed
- if `score < acceptanceScore` and iterations remain, improve and re-vet
- if iteration 5 is reached and `score >= goodEnoughScore`, mark `good enough` and proceed
- if iteration 5 is reached and `score < goodEnoughScore`, stop and escalate for human review

This prevents infinite loops while still allowing bounded pragmatism.

### Stage 8: execute

Only `accepted` or `good enough` plans run.

Execution must occur in an isolated git worktree on a feature branch. The worktree path and branch name are part of persisted work state so the run can resume cleanly without tying up the primary working tree.

### Stage 9: close

Persist:

- plan path
- source issue linkage
- iteration count
- scores and findings
- chosen reviewers
- worktree path
- feature branch
- execution outcome
- timestamps for ranking and resume

## Review Standard

The design should avoid fuzzy standards like "would Claude with superpowers think this is good?" and replace them with an explicit rubric such as `principal_engineer_plan_review`.

### Reviewer profiles

Reviewer selection is driven by `franken-skills` metadata and touched-file/domain hints.

Initial reviewer set:

- principal TypeScript engineer
- principal CLI/orchestrator engineer
- staff test/reliability engineer
- security/governance reviewer

### Score categories

- completeness
- chunkability
- dependency correctness
- verification quality
- codebase alignment
- execution readiness
- rollback and safety
- test adequacy

### Thresholds

The system must expose two thresholds:

- `acceptanceScore`: normal runnable threshold
- `goodEnoughScore`: lower fallback threshold used only at iteration cap

The workflow should record both the composite score and the reviewer-specific breakdown so decisions remain auditable.

## Plan Preparation Command

`frankenbeast plan prepare <paths...>` exists to make manual plan authoring a supported workflow, not a hidden hack.

Rules:

- if target is a file, patch that file in place
- if target is a directory, recursively patch Markdown files inside it
- preserve unrelated frontmatter keys
- append missing required keys to existing frontmatter
- create frontmatter when none exists
- when a file appears to be a chunk, require or safely infer `implementationPlan`
- if lineage cannot be inferred safely, report and skip

This should rely on the same frontmatter parser and serializer used by `work` discovery so the system has one artifact truth.

## Documentation Changes

This feature is not complete without user-facing docs.

Required docs updates:

- `README.md` for the new `work` and `plan prepare` commands
- `docs/guides/fix-github-issues.md` to reframe `issues` as a `work` source
- `docs/guides/quickstart.md` to teach the end-to-end workflow
- new guide for authoring implementation plans and chunks manually

The docs must explain enough that a user can create a compliant implementation plan by hand and have `frankenbeast work` discover it on the first try.

## Testing Strategy

Required coverage:

- frontmatter parsing and serialization
- parent plan discovery
- chunk exclusion from top-level execution
- plan ranking by priority/status/freshness
- `plan prepare` merge behavior
- planner/chunk writers stamping required frontmatter
- `work issues` label filtering and default priority order
- issue freshness rejection for already-fixed work
- review loop thresholds and iteration cap
- `good enough` fallback behavior
- worktree creation, branch naming, and state persistence
- CLI alias behavior for `issues`

## Risks

### Over-discovering stray Markdown

Mitigation:

- only treat files with explicit frontmatter as runnable plans
- ignore chunks and ignored directories

### Review loops becoming expensive or noisy

Mitigation:

- hard cap at 5 iterations
- structured reviewer selection
- persisted scores and findings for resume

### Unsafe frontmatter retrofits

Mitigation:

- conservative `plan prepare`
- skip ambiguous chunk lineage rather than guessing

### Worktree sprawl

Mitigation:

- persist worktree metadata
- add cleanup guidance and follow-up automation later

## Migration

This design is backward-compatible with existing commands:

- `interview`, `plan`, and `run` remain
- `issues` remains as an alias
- existing chunk files continue to work for lower-level execution flows

The new behavior layers a canonical artifact contract and a top-level workflow on top of existing primitives rather than deleting them.
