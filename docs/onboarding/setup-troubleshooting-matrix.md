# Setup troubleshooting matrix

Use this matrix when a first-run setup, local bootstrap, or issue-worker preflight fails. Start with the symptom that matches the first failing command, collect the diagnostic evidence, apply the smallest safe remediation, then run the verification command before continuing.

Avoid destructive cleanup while diagnosing. Rows that mention deleting generated state require a backup or a move to a timestamped quarantine path first.

| Symptom | Likely cause | Diagnostic command | Remediation | Verification command |
| --- | --- | --- | --- | --- |
| `npm install`, `npm ci`, or `npm run bootstrap` fails with an engine error | Node.js does not satisfy the root `engines.node` range or `engine-strict=true` is rejecting the active runtime | `node --version && node -p "require('./package.json').engines.node"` | Switch to a supported Node.js version from `.nvmrc` or the root `engines.node` range, then open a new shell so `node` and `npm` resolve to the same toolchain | `node -p "process.version" && npm --version && npm run check:package-manager` |
| `npm run check:package-manager` reports a package-manager mismatch | Corepack is disabled, the npm shim is stale, or a different package manager is active | `node -p "require('./package.json').packageManager" && npm --version && command -v corepack` | Enable Corepack for npm and activate the exact pinned package manager: `corepack enable npm && corepack prepare "$(node -p "require('./package.json').packageManager")" --activate` | `npm --version && npm run check:package-manager` |
| `corepack: command not found` before bootstrap | The installed Node.js distribution does not bundle Corepack or it was not installed on PATH | `node --version && command -v corepack || true` | Install Corepack for the active Node.js runtime (`npm install -g corepack`), then run the repository Corepack activation commands from `ONBOARDING.md` | `command -v corepack && corepack --version && npm run check:package-manager` |
| `gh auth status` fails during `new-worker:preflight`, issue worktree bootstrap, or PR creation | GitHub CLI is not installed, is not logged in, or is logged in without repository write access | `command -v gh && gh auth status && gh repo view djm204/frankenbeast --json viewerPermission` | Install/log in with `gh auth login` using an account that can read the repository and, for PR-producing work, write branches/comments | `gh auth status && gh repo view djm204/frankenbeast --json viewerPermission` |
| Bootstrap or a dev server reports `EADDRINUSE` or a dashboard/chat port is already in use | A prior local server, dashboard, chat backend, or test process is still listening on the configured port | `for port in ${FAILED_PORTS:-5173 3737}; do PORT="$port" node -e "const port=Number(process.env.PORT); import('node:net').then(({createServer}) => { const s=createServer().once('error', e => { console.error(port + ': ' + e.code); process.exitCode=1; }).once('listening', () => s.close(() => console.log(port + ': free'))).listen(port, '127.0.0.1'); })"; done` | Stop the owning process from the previous terminal/session, or choose an explicit unused port for the new server; do not kill unrelated system processes without confirming the owner | Rerun the same port probe with `FAILED_PORTS=<failed-port>` or the default `5173 3737`, then restart the failed command |
| `new-worker:preflight` reports a dirty worktree before coding | Generated files, previous worker edits, or scratch artifacts are present in the issue checkout | `git status --short --branch && git diff --stat` | Use a fresh isolated worktree for the issue. If generated artifacts must be removed, first move them to a timestamped backup/quarantine directory outside the worktree or confirm they are ignored scratch files | `git status --short --branch` |
| Bootstrap or runtime commands fail on `.fbeast/*.lock`, SQLite busy, or stale lock-file errors | A prior process exited while holding a generated lock or a live Frankenbeast process is still using local state | `ps -ef \| grep -E 'frankenbeast\|fbeast\|node .*franken' \| grep -v grep \|\| true; find .fbeast -maxdepth 2 -type f -name '*.lock' -print` | First stop live Frankenbeast processes cleanly. If no process owns the state, back up `.fbeast` or move the lock file to a timestamped quarantine path before retrying; do not delete the state directory wholesale | `npm run local:verify-setup` or the originally failed command |
| `npm --silent run new-worker:preflight -- --json` returns failed checks | Required tools, git identity, GitHub auth, repository root, or worktree cleanliness does not match issue-worker expectations | `npm --silent run new-worker:preflight -- --json` | Fix each `fail` check directly: install missing `git`/`gh`/`jq`, log in to GitHub, set `git config user.name 'David Mendez'` and `git config user.email 'me@davidmendez.dev'`, or move to a clean issue worktree | `npm --silent run new-worker:preflight -- --json` |
| Docker-backed optional services fail during `npm run bootstrap -- --services` | Docker is not running, credentials are missing, or optional ChromaDB/Grafana/Tempo containers are unhealthy | `docker compose ps; docker compose logs --tail=80 chromadb grafana tempo` | Start Docker, set required local service env such as a unique `GRAFANA_PASSWORD`, and restart the required optional service set. Unit tests and most docs changes can continue with `--no-docker` | For the full optional stack, run `docker compose ps && npm run local:verify-setup`; for a partial stack, use targeted probes such as `curl -fsS "$CHROMA_URL/api/v2/heartbeat"`, `curl -fsS http://localhost:3000/api/health`, or `curl -fsS http://localhost:3200/ready` for only the services you started |
| Provider, dashboard, or chat setup fails with missing token/secret errors | Runtime environment and configured `network.secureBackend` do not contain the same operator/provider secrets | `node packages/franken-orchestrator/dist/cli/run.js init --help; test -f .fbeast/config.json && node -e "console.log(JSON.parse(require('node:fs').readFileSync('.fbeast/config.json','utf8')).network?.secureBackend)"` | Choose the secret backend before init, authenticate that backend (`op`, `bw`, OS keychain, or local encrypted passphrase), then re-store or migrate existing secret refs when changing backends | Rerun the originally failed command, or use `frankenbeast network status` / `node packages/franken-orchestrator/dist/cli/run.js network status` to confirm refs are configured; run `init --repair` only when you intentionally want an interactive repair that may update local state |

## Evidence bundle for handoffs

When escalating a setup failure to a PM, maintainer, or follow-up agent, include:

- the exact failed command and full error text;
- output from `node --version`, `npm --version`, `npm run check:package-manager`, and `npm --silent run new-worker:preflight -- --json` when relevant;
- `git status --short --branch` from the failing checkout;
- the selected optional-service mode (`--no-docker` or `--services`) and `docker compose ps` output when Docker services are involved;
- confirmation that any generated-state cleanup was backed up or quarantined before retrying.
