# Live CLI Benchmark Pipeline Design

**Date:** 2026-04-26
**Status:** Approved
**Scope:** Recurring real-client benchmark pipeline for `Codex CLI` and `Gemini CLI`, comparing baseline runs against `@fbeast/mcp-suite` runs over time.

## Goal

Build a recurring benchmark pipeline that proves how Frankenbeast changes real client behavior when the same tasks are run:

1. without Frankenbeast
2. with `@fbeast/mcp-suite` installed

The benchmark must use real `Codex CLI` and real `Gemini CLI`, not a synthetic harness that simulates those clients. The result should be a defensible A/B dataset that tracks accuracy, latency, token/cost behavior, and workflow quality over time.

## Success Criteria

The benchmark is successful when it can:

- run the same versioned task corpus on `Codex CLI` and `Gemini CLI`
- run each task in both `baseline` and `fbeast` modes under the same conditions
- capture enough evidence to explain both successful and failed runs
- score results with deterministic checks first and optional judge-based checks second
- persist normalized benchmark history for trend analysis
- flag regressions over time and support release-gate decisions on a stable core corpus

## Product Boundary

This system is a benchmark runner above `@fbeast/mcp-suite`, not a benchmark implementation inside the suite itself.

Responsibilities of the benchmark runner:

- define the corpus
- provision isolated workspaces
- configure clients for baseline versus `fbeast`
- launch real CLI runs
- collect raw evidence
- score results
- persist normalized history
- generate reports and gate decisions

Non-responsibilities of `@fbeast/mcp-suite`:

- it should not own benchmark orchestration
- it should not accumulate benchmark-specific storage concerns
- it should not be modified solely to host reporting logic that belongs to the benchmark layer

The suite remains the system under test.

## Client Scope

The recurring benchmark supports only these live clients:

- `Codex CLI`
- `Gemini CLI`

Excluded from this benchmark:

- provider-only API replays
- simulated client loops
- synthetic microbenchmarks presented as primary evidence

Microbenchmarks may still exist as secondary diagnostics, but the benchmark record of truth is the real-client run.

## Benchmark Matrix

Each benchmark row is defined by:

- `client`: `codex-cli` or `gemini-cli`
- `mode`: `baseline` or `fbeast`
- `fbeast_topology`: `none`, `proxy`, or `split`
- `project_fixture`
- `task_id`
- `model`
- `client_version`
- `commit_sha`
- `run_timestamp`
- `host_class`

The benchmark is an A/B matrix, not a single scorecard.

Required runs per task:

1. `Codex CLI` baseline
2. `Codex CLI` + Frankenbeast
3. `Gemini CLI` baseline
4. `Gemini CLI` + Frankenbeast

### Fairness Rules

For every baseline versus `fbeast` comparison, the following must remain fixed:

- same project snapshot
- same task prompt
- same client version
- same model configuration
- same timeout budget
- same retry policy
- same machine class

Only the Frankenbeast installation state may change between the A and B run.

### Topology Rule

`baseline` means no Frankenbeast at all:

- no `fbeast` MCP registration
- no `fbeast` hooks
- no `.fbeast` runtime generated by the benchmark setup unless a task explicitly creates it

`fbeast` means the same client and task with Frankenbeast intentionally installed and configured.

Frankenbeast topology remains an explicit dimension:

- `proxy`
- `split`

This allows the benchmark to answer both:

- whether Frankenbeast helps or hurts relative to baseline
- whether proxy mode behaves differently from split-server mode

## Execution Model

Each task execution should be isolated and reproducible.

Per run, the benchmark runner must:

1. create a fresh workspace from a versioned fixture
2. prepare client-specific config for either `baseline` or `fbeast`
3. install `@fbeast/mcp-suite` only for `fbeast` runs
4. launch the real client in a scripted non-interactive mode
5. wait for completion, timeout, or explicit failure
6. capture raw outputs, artifacts, and telemetry
7. score the run against the task rubric
8. persist both raw evidence and normalized metrics

### Client Adapters

The benchmark runner should define one executor adapter per client:

- `codex-cli` executor
- `gemini-cli` executor

Each adapter is responsible for:

- writing the client-specific config needed for the run
- launching the client process
- capturing stdout, stderr, exit code, and transcript data
- exposing any client-native telemetry that can be collected reliably

The benchmark core should not contain client-specific shell behavior beyond those adapters.

## Corpus Design

The corpus must be fixed, versioned, and designed to expose differences Frankenbeast is expected to cause.

### Task Classes

The corpus should contain three classes of tasks:

- `tool-critical`: success depends on correct tool discovery and argument selection
- `workflow-critical`: success depends on multi-step ordering, retries, or guardrail behavior
- `artifact-critical`: success leaves behind objectively checkable files or output

### Task Definition Contract

Each task fixture must define:

- `task_id`
- `project_fixture`
- `prompt`
- `expected_artifacts`
- `required_checks`
- `timeout_ms`
- `allowed_nondeterminism`
- `baseline_supported`
- `notes`

`baseline_supported` is required because some tasks may be unfair or meaningless without Frankenbeast. Those tasks may remain in exploratory suites, but they must not be used for baseline-versus-fbeast core gating if the comparison is not legitimate.

### Corpus Tiers

The corpus should be split into:

- `core`: stable, versioned tasks used for trend lines and release gates
- `candidate`: new tasks being validated before they influence long-term trends
- `stress`: longer or noisier tasks used for exploratory benchmarking, not release gates

This prevents metric drift from constant corpus churn.

## Metrics Model

Every run should emit four metric groups.

### 1. Outcome Metrics

- `task_pass`
- `accuracy_score`
- `rubric_failures`
- `artifact_diff`

These are the primary benchmark outputs. If Frankenbeast improves safety but reduces task completion, the benchmark must show that directly.

### 2. Speed Metrics

- `wall_clock_ms`
- `time_to_first_tool_ms` when observable
- `time_to_last_output_ms`
- `tool_count`
- `tool_round_trips`

These quantify the overhead or efficiency change introduced by Frankenbeast.

### 3. Cost Metrics

- `client_reported_prompt_tokens`
- `client_reported_completion_tokens`
- `client_reported_cost_usd`
- `fbeast_observed_prompt_tokens`
- `fbeast_observed_completion_tokens`
- `fbeast_observed_cost_usd`
- `unattributed_cost_flag`

Reports must not flatten these into one misleading number.

The benchmark must distinguish:

- cost reported by the client
- cost observed by Frankenbeast tooling
- any unknown or unattributed gap

### 4. Behavior Metrics

- `expected_tool_usage_score`
- `hallucinated_tool_call_count`
- `golden_trace_score`
- `policy_intervention_count`
- `error_count`
- `timeout`

These metrics exist to prove workflow changes rather than only cost changes.

## Scoring Model

Deterministic scoring is primary. Judge-based scoring is secondary.

### Deterministic First

Primary checks should include:

- exact artifact checks for file-creating or file-editing tasks
- tool-call schema checks
- golden trace comparisons for workflow-shaped tasks
- explicit rubric assertions per task

Relevant existing building blocks already exist in this repo:

- `ToolCallAccuracyEval` in `packages/franken-observer/src/evals/deterministic/ToolCallAccuracy.ts`
- `GoldenTraceEval` in `packages/franken-observer/src/evals/regression/GoldenTraceEval.ts`
- `fbeast_observer_log_cost` and session cost aggregation in `packages/franken-mcp-suite`

### Secondary Judge Score

An optional LLM judge score may be stored for qualitative trend analysis, but it must not be the primary pass/fail gate.

Judge output is appropriate for:

- qualitative ranking among close variants
- human-readable explanations
- exploratory trend analysis

Judge output is not appropriate for:

- authoritative pass/fail gating
- replacing deterministic artifact checks

### Pass Rule

Recommended pass rule:

- `task_pass = true` only if all required deterministic checks pass
- `accuracy_score` is a weighted deterministic aggregate
- `judge_score` is stored separately and reported as advisory

## Evidence Collection

The benchmark must capture enough raw evidence to explain the normalized result later.

Each run should retain:

- client stdout
- client stderr
- exit code
- transcript or response log when available
- generated artifacts
- task rubric outputs
- environment manifest
- run timestamps
- timeout and retry outcomes

For Frankenbeast runs, evidence should also include:

- `.fbeast/beast.db`
- resolved Frankenbeast config
- installed topology
- observer cost summaries
- audit trail extracts

This separation matters because baseline runs will not produce `.fbeast` evidence and should not be forced into that shape.

## Storage Model

Do not use `.fbeast/beast.db` as the benchmark warehouse.

That database is per-run evidence generated by the system under test, not the long-term benchmark history.

Use two storage layers.

### Run-Local Evidence

Store raw evidence on disk under a run artifact root such as:

`benchmarks/runs/YYYY-MM-DD/<run-id>/`

This directory should contain:

- workspace metadata
- raw outputs
- artifacts
- diffs
- environment manifest
- optional copied `.fbeast/` state for Frankenbeast runs

### Benchmark Warehouse

Store normalized history in a dedicated benchmark SQLite database such as:

`benchmarks/history/benchmarks.db`

This warehouse should be append-only and contain:

- one row per run
- child tables for metric groups
- check results
- tool-call summaries
- artifact summaries
- A/B pairing metadata

The warehouse becomes the source of truth for:

- trend charts
- regression detection
- release gates
- historical comparisons

## Reporting Model

Every recurring benchmark execution should emit:

- per-task A/B comparison
- per-client summary
- trend versus the last `N` runs
- regression flags
- improvement flags

The report should answer:

- does Frankenbeast improve task success?
- what latency overhead does it add?
- what cost delta does it add or save?
- does it reduce bad tool behavior or unstable workflows?
- are those effects stable over time?

## Release-Gate Policy

Only the stable `core` corpus may drive release gates.

Recommended gate policy:

- fail when `fbeast` accuracy regresses beyond threshold against its own recent benchmark baseline
- fail when `fbeast` loses to `baseline` on core task pass rate by more than threshold
- warn rather than fail on latency or cost regressions unless they cross an explicit hard ceiling
- never gate on judge-only metrics

This keeps the release signal tied to user-visible correctness first.

## Integration With Existing Repo Primitives

The benchmark should reuse existing repo capabilities where possible instead of inventing parallel telemetry:

- `packages/franken-mcp-suite/src/servers/observer.ts`
  This already exposes `fbeast_observer_log_cost`, cost summaries, and audit-trail access.
- `packages/franken-mcp-suite/src/shared/sqlite-store.ts`
  This already defines the run-local observer and cost tables inside `.fbeast/beast.db`.
- `packages/franken-observer/src/evals/deterministic/ToolCallAccuracy.ts`
  This provides deterministic tool-call scoring logic.
- `packages/franken-observer/src/evals/regression/GoldenTraceEval.ts`
  This provides deterministic workflow regression scoring against a golden trace.

The benchmark runner should wrap these primitives rather than duplicate them.

## Non-Goals

This design does not attempt to:

- benchmark Claude Code in this pipeline
- make different clients fully deterministic
- force client-native telemetry into a fake unified cost number
- use judge-only scoring as release authority
- fold benchmark history into the product runtime database

## Risks And Mitigations

### Client telemetry asymmetry

Risk:
`Codex CLI` and `Gemini CLI` may expose different levels of token and transcript fidelity.

Mitigation:
Keep separate fields for client-reported and Frankenbeast-observed cost, and explicitly track unattributed gaps.

### Nondeterministic live-model behavior

Risk:
Real-client runs will vary over time.

Mitigation:
Use repeated runs where needed, keep the core corpus stable, and gate only on stable deterministic signals with explicit thresholds.

### Unfair baseline comparisons

Risk:
Some tasks may implicitly depend on Frankenbeast-only affordances.

Mitigation:
Mark tasks with `baseline_supported` and exclude unfair comparisons from the core release-gate set.

### Benchmark contamination

Risk:
Workspace state or client config leaks between runs and pollutes A/B results.

Mitigation:
Create fresh workspaces and isolated client config per run, and persist environment manifests for audit.
